RateLock optimizes each storage engine natively to achieve high throughput and low latency. Instead of wrapping queries in a generic database translation layer, the library targets native database features. Redis uses Lua scripts for atomic operations, PostgreSQL uses isolated UPSERT queries, and the local memory engine uses JavaScript Maps.

This page shows performance data from our standardized benchmark suite under concurrent workloads.

---

## Performance Summary

These metrics reflect concurrent rate-limiting operations:

- **In-Memory Throughput**: Memory counter strategies handle between 1.2M and 2.5M operations per second per node depending on key distribution.
- **Competitor Comparison**: RateLock Fixed Window throughput is higher than `rate-limiter-flexible` (3.6x in memory, 5.6x over Redis).
- **Spam Mitigation**: The `withCache` decorator shields backend databases by intercepting repeat blocked requests in memory, scaling spam-check throughput to 2.4M operations per second.
- **Postgres Optimization**: Unlogged tables in PostgreSQL improve counter write performance by approximately 18% compared to standard logged tables.

## 1. Local Memory Strategy Trade-offs

Rate-limiting strategies balance mathematical precision against CPU overhead.

| Strategy             | Throughput (Ops/sec) | Average Latency | Algorithmic Complexity            |
| :------------------- | :------------------- | :-------------- | :-------------------------------- |
| **Fixed Window**     | **2,416,711**        | 0.03ms          | $O(1)$ Counter increments         |
| **Token Bucket**     | **2,374,650**        | 0.03ms          | $O(1)$ Time-based math            |
| **Individual Fixed** | **2,502,058**        | 0.03ms          | $O(1)$ Dynamic counters           |
| **Sliding Window**   | **58,799**           | 1.36ms          | $O(N)$ Array timestamp operations |

> [!IMPORTANT]
> **Sliding Window Array Overhead**
> The Sliding Window strategy checks a moving window at millisecond precision. To do this, the engine must store a timestamp for every request in an array or a Redis Sorted Set. On every check, it filters out expired timestamps.
>
> This array allocation and filtering runs in $O(N)$ time. Counter-based strategies like Fixed Window or Token Bucket increment a single number in $O(1)$ time. This difference explains why the Sliding Window strategy handles fewer operations (~60,000 ops/sec) than Token Bucket (~2.4 Million ops/sec). For most high-throughput applications, Token Bucket offers the best balance of precision and performance.

## 2. Competitive Analysis: RateLock vs Competitors

We compared RateLock against `rate-limiter-flexible` (RLF) in a high-concurrency spam scenario. We ran 80 parallel workers querying a single key.

### In-Memory Counter Engines

RateLock uses direct key lookups in a native JavaScript `Map` to avoid object allocation.

- **RateLock Local Fixed Window**: **2,166,631 ops/sec** (Avg Latency: 0.04ms)
- **rate-limiter-flexible Memory**: **594,228 ops/sec** (Avg Latency: 0.13ms)
- **Throughput ratio**: RateLock handles 3.6x more operations.

---

### Redis Distributed Engines

RateLock executes a single Lua script for Redis checks. The script runs atomically on the Redis thread, returning the limit decision in a single round-trip. Competitors often perform multiple Redis queries or wrap results in complex client objects, increasing network overhead.

- **RateLock Redis Fixed Window**: **131,650 ops/sec** (Avg Latency: 0.61ms)
- **rate-limiter-flexible Redis**: **23,221 ops/sec** (Avg Latency: 3.35ms)
- **Throughput ratio**: RateLock handles 5.6x more operations.

> [!NOTE]
> **Strategy Availability**
> RLF implements only the Fixed Window strategy for database-backed engines (Redis and Postgres). RateLock implements all four strategies (Fixed Window, Sliding Window, Token Bucket, and Individual Fixed Window) natively on all three backends.

### PostgreSQL: Logged vs Unlogged Tables

Write logging directly affects PostgreSQL rate-limiting throughput.

- **node-postgres - Fixed Window (Unlogged)**: **7,723 ops/sec** (Avg Latency: 10.3ms)
- **node-postgres - Fixed Window (Logged)**: **6,539 ops/sec** (Avg Latency: 12.1ms)
- **Performance difference**: Unlogged tables increase throughput by 18%.

> [!TIP]
> **Why Unlogged Tables Work**
> Creating rate-limiting tables as `UNLOGGED` tells PostgreSQL to skip writing transactions to the Write-Ahead Log (WAL). If the database crashes, Postgres truncates these tables on recovery. This truncation is acceptable for transient rate-limiting counters and improves SSD longevity.

## 4. Resilience Policies (Decorators Overhead)

You can wrap rate limiters in resilience decorators (`withCache`, `withCircuitBreaker`, `withFallback`, `withRetry`). These decorators add negligible CPU overhead. In spam scenarios, the `withCache` decorator increases overall throughput by intercepting requests in memory.

| Limiter Layer Configuration               | Throughput (Ops/sec) | Latency Overhead       |
| :---------------------------------------- | :------------------- | :--------------------- |
| **Raw Local Fixed Window**                | **1,088,680**        | Reference              |
| **Fixed Window + withCircuitBreaker**     | **1,195,936**        | < 1% difference        |
| **Fixed Window + withFallback**           | **1,115,179**        | < 1% difference        |
| **Fixed Window + withRetry**              | **828,946**          | Under retry simulation |
| **Fixed Window + withCache (Deny Cache)** | **2,418,094**        | 2.2x Increase          |

### Intercepting Spam in Memory (`withCache`)

When you wrap a backend limiter (Postgres or Redis) in the `withCache` policy, RateLock stores blocked keys in a local, short-lived memory cache. When a client exceeds their limit, the first blocked request populates this cache. Subsequent requests from that client fail immediately in memory, preventing network round-trips to Redis or PostgreSQL.

- **Without Deny Cache**: Spammed requests query Postgres directly, capping at 1,200 ops/sec.
- **With Deny Cache**: Spammed requests fail in memory at 2,418,094 ops/sec.
- **Result**: This mechanism shields Postgres and Redis clusters from database load during high-volume spam.

## 5. Run the Benchmarks Locally

You can execute these benchmarks locally. The test suite manages Postgres, Redis, and Valkey containers via Docker, executes the test matrix, compiles the reports, and stops the containers.

```bash
# Clone the repository
git clone https://github.com/saoudi-h/ratelock.git
cd ratelock

# Install dependencies
pnpm install

# Run the standardized benchmark suite
pnpm --filter @ratelock/bench bench:full
```
