---
title: Standalone Decorators
description: Wrap existing limiters with additional behavior for advanced use cases.
---

> **Note:** Most users should use the [built-in decorator options](/docs/adapters) at limiter creation time. Standalone decorators are for advanced scenarios where you need flexibility beyond the defaults.

## Built-in vs Standalone

RateLock offers two ways to add decorators to a limiter:

### Built-in (recommended)

Applied at creation time, fixed for the lifetime of the limiter:

```typescript
const limiter = await createFixedWindowLimiter({
    url: 'redis://localhost:6379',
    limit: 100,
    windowMs: 60_000,
    cache: { maxSize: 1000, ttlMs: 30_000 },
    retry: { maxAttempts: 3 },
})
```

**Pros:** Simple, one place to configure, no wrapping overhead.
**Cons:** Fixed for the lifetime of the limiter.

### Standalone (advanced)

Wrap an existing limiter at any point in your code:

```typescript
import { withCache, withRetry } from '@ratelock/redis'

const limiter = await createFixedWindowLimiter({ ... })
const wrapped = withCache(withRetry(limiter, { maxAttempts: 3 }), { maxSize: 500 })
```

**Pros:** Flexible, can vary by context, full control over composition order.
**Cons:** More verbose, risk of double-wrapping if combined with built-in options.

## When to Use Standalone

### Conditional Decorators by Route

Apply different decorators to the same limiter depending on the route:

```typescript
import { createSlidingWindowLimiter, withCache } from '@ratelock/redis'

const limiter = await createSlidingWindowLimiter({
    url: 'redis://localhost:6379',
    limit: 100,
    windowMs: 60_000,
})

// Public endpoints: add cache
const publicLimiter = withCache(limiter, { maxSize: 1000, ttlMs: 30_000 })

// Auth endpoints: no cache, always check Redis
const authLimiter = limiter
```

### Contextual Retry

Retry only for critical operations:

```typescript
import { withRetry } from '@ratelock/redis'

const limiter = await createFixedWindowLimiter({ ... })

// Payment endpoint: retry on transient failures
const paymentLimiter = withRetry(limiter, { maxAttempts: 3 })

// Health check: no retry, fail fast
const healthLimiter = limiter
```

### Testing

Simulate backend failures without changing production code:

```typescript
import { withErrorPolicy } from '@ratelock/redis'

const limiter = await createFixedWindowLimiter({ ... })

// In tests: fail-open to simulate Redis being down
const testLimiter = withErrorPolicy(limiter, 'allow')
```

### Dynamic Composition Order

Control the order of decorator application:

```typescript
import { withCache, withRetry, withCircuitBreaker } from '@ratelock/redis'

const limiter = await createFixedWindowLimiter({ ... })

// Cache first, then retry (retry only on cache miss)
const v1 = withRetry(withCache(limiter, cacheConfig), retryConfig)

// Retry first, then cache (cache the result after retries)
const v2 = withCache(withRetry(limiter, retryConfig), cacheConfig)
```

## Available Decorators

All decorators are re-exported from each adapter package:

```typescript
import { withCache, withRetry, withCircuitBreaker, withErrorPolicy } from '@ratelock/local'
// or @ratelock/redis, @ratelock/postgres
```

| Decorator | Purpose |
|-----------|---------|
| `withCache` | Cache denied results to reduce backend load |
| `withRetry` | Retry transient failures with exponential backoff |
| `withCircuitBreaker` | Stop calling a failing backend temporarily |
| `withErrorPolicy` | Define behavior when the limiter throws |

## Composition Order

Decorators are applied from outside in. The outermost decorator runs first:

```
withErrorPolicy(          ← catches errors from everything below
  withCircuitBreaker(     ← stops calling if too many failures
    withRetry(            ← retries on transient errors
      withCache(          ← caches denied results
        limiter           ← the actual rate limiter
      )
    )
  )
)
```

**Recommended order:** `withErrorPolicy` → `withCircuitBreaker` → `withRetry` → `withCache` → `limiter`

## ⚠️ Don't Stack Built-in and Standalone

If you use built-in decorator options at creation time, avoid wrapping the same limiter with a standalone decorator of the same type:

```typescript
// ❌ Don't do this — double cache with different configs
const limiter = await createFixedWindowLimiter({
    url: 'redis://...',
    cache: { maxSize: 1000, ttlMs: 30_000 },  // built-in cache
})

const wrapped = withCache(limiter, { maxSize: 500, ttlMs: 10_000 }) // standalone cache on top
```

**What happens:** Both caches run, resulting in redundant Map lookups and conflicting TTLs. The behavior is predictable (outer cache checks first) but wasteful and confusing.

**Rule:** Choose one approach per decorator type — either built-in OR standalone, not both.
