---
title: withCache
description: Cache denied results to reduce backend load.
---

The `withCache` decorator caches **denied** (rate-limited) results so subsequent checks for the same identifier return immediately without hitting the backend.

## Usage

```typescript
import { withCache } from '@ratelock/redis'

const limiter = await createFixedWindowLimiter({ ... })
const cached = withCache(limiter, {
    maxSize: 1000,   // max cached entries
    ttlMs: 30_000,   // cache entries expire after 30 seconds
})
```

| Option | Type | Description |
|--------|------|-------------|
| `maxSize` | `number` | Maximum number of cached entries (LRU eviction) |
| `ttlMs` | `number` | Time-to-live for cached entries in milliseconds |

## How It Works

```
check('user:123') → denied → cache it
check('user:123') → return cached denied (no backend call)
check('user:123') → cache expired → check backend again
```

Only denied results are cached. Allowed requests always go through to the backend.

## Built-in Alternative

For most cases, use the built-in `cache` option at creation time instead:

```typescript
const limiter = await createFixedWindowLimiter({
    url: 'redis://localhost:6379',
    limit: 100,
    windowMs: 60_000,
    cache: { maxSize: 1000, ttlMs: 30_000 },
})
```

See [Standalone Decorators](/docs/decorators) for when to use standalone vs built-in.
